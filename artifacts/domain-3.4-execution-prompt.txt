--- DECISIONS ---

D1 — How to fix the Search Law violation in organizationsMapData.ts:
Refactor `loadOrganizationsMapRows()` to query `provider_search_index` instead of `organizations` table, using `querySearchIndex()` from lib/server/search/searchRepository.ts. Add `address`, `phone`, `website` columns to provider_search_index via migration and populate them from `organizations.metadata` in `syncOrgToIndex()`. Map SearchResult rows to OrganizationMapRow shape using `state_codes` for region_label (no county data in index — use `regionLabelForOrg(state_codes, [])`). External CBO rows (from JSON file) are merged in after the index query — unchanged.

D2 — Visibility rule after migration:
D3.4 uses `is_active=true` from provider_search_index as the eligibility gate (equivalent to canOrganizationAppearInSearch() passing). Orgs with lifecycle_status=seeded or draft profiles will NOT appear on the discovery map until they complete onboarding. This is the correct architectural decision — the search index is the single source of truth for discovery. Document the visibility change in DEFERRED log as [DEFERRED-3.4-001].

D3 — syncOrgToIndex wiring location:
Wire `syncOrgToIndex()` as a fire-and-forget call in `app/api/org/profile/route.ts` POST handler, after the profile update succeeds and logEvent fires. Pattern: `void syncOrgToIndex({ organizationId: rowId }, getSupabaseAdmin()).catch((err) => logger.warn("syncOrgToIndex.error", { error: err.message }));` Do NOT await it — it must not block the HTTP response.

D4 — Policy action for provider browsing:
Add `provider_search:browse` to actionRegistry.ts. Policy evaluation: any authenticated user (applicant, provider, platform_admin) may browse providers. Unauthenticated requests are denied. The map routes will use `can("provider_search:browse", actor, { type: "provider_search", id: "all", ownerId: "" })` as the auth gate. Remove `requireRole` calls from map routes (they will no longer be needed as the policy action covers all valid authenticated callers).

D5 — Where in policyEngine.ts does provider_search:browse live:
Add a new `evalProviderSearch` function and dispatch to it for `provider_search:*` resource types. OR: add a simple case in the top-level can() switch. Given the action is simple (any authenticated user), add it as a case in the existing switch dispatch in policyEngine.ts — no new evaluator file needed. Decision: handle inline in the existing evaluator dispatch for simplicity.

D6 — What to do with the advocate organizations-map route:
Both `/api/victim/organizations-map` and `/api/advocate/organizations-map` call `loadOrganizationsMapRows()`. Both are refactored to use the updated function (after it reads from provider_search_index). Both get `can("provider_search:browse", ...)` gates. The language "victim" in the route path is NOT changed.

D7 — OrganizationMapRow shape compatibility:
The existing `OrganizationMapRow` type in organizationsMapData.ts has `region_label: string` and `states: string[]`. When reading from provider_search_index, use `state_codes` for `states` and compute `region_label` with `regionLabelForOrg(state_codes, [])`. The `response_accessibility` field cannot be populated from the index (it requires org-specific data). Set `response_accessibility: null` for all index rows — this data is not available in the search layer without an org join. Document as [DEFERRED-3.4-002].

D8 — No new API routes:
D3.4 does NOT create new API routes. It refactors existing ones. No `GET /api/providers/search` is created in this domain. That is a future domain.

New implementation:

```typescript
import { querySearchIndex } from "@/lib/server/search/searchRepository";
import { getSupabaseAdmin } from "@/lib/supabaseAdmin";

export async function loadOrganizationsMapRows(): Promise<OrganizationMapRow[]> {
  const supabase = getSupabaseAdmin();
  // Search Law: read from provider_search_index only — no direct organizations query.
  // Select all active providers with no text/geo filter.
  const { data, error } = await supabase
    .from("provider_search_index")
    .select("org_id,name,service_tags,state_codes,accepting_clients,capacity_status,approximate,lat,lng,address,phone,website")
    .eq("is_active", true)
    .limit(500);

  if (error) throw new Error(error.message);

  const dbRows: OrganizationMapRow[] = (data ?? []).map((row) => ({
    id: row.org_id,
    name: row.name,
    lat: row.lat ?? 0,
    lng: row.lng ?? 0,
    approximate: Boolean(row.approximate),
    accepting_clients: Boolean(row.accepting_clients),
    capacity_status: row.capacity_status ?? "unknown",
    region_label: regionLabelForOrg(Array.isArray(row.state_codes) ? row.state_codes : [], []),
    states: Array.isArray(row.state_codes) ? row.state_codes : [],
    address: row.address ?? null,
    phone: row.phone ?? null,
    website: row.website ?? null,
    response_accessibility: null, // [DEFERRED-3.4-002]: not available in search index
  }));

  const cboRows = loadCboVaMapRows();
  return [...dbRows, ...cboRows];
}
```
