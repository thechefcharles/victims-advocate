Routes become thin orchestrators: parse body → call service → serialize → return. Rationale: Law 4 prohibits business logic in routes; no service currently exists.

D5 — Public program listing (Q5):
NO public program listing endpoint in this domain. Programs remain admin-only for D3.3. A public endpoint is deferred. Rationale: no product requirement for public discovery is scoped here; adding an unscoped endpoint would exceed boundary.

The partial unique index already exists in the DB (`UNIQUE on program_key WHERE is_active=true`). No new migration is needed for this constraint. `programService.ts` must also enforce this at the service layer with an explicit check before activation: query for any existing `is_active=true` record with the same `program_key`, deactivate it, then activate the target. Dual enforcement (DB + service) is the correct pattern.

## CREATE: lib/server/programs/programService.ts

Purpose: Single service owning all program definition mutations and queries.

Exports:
- `listProgramDefinitions(filters?: { status?: string; state_code?: string }): Promise<ProgramDefinitionRow[]>`
- `createProgramDefinition(payload: CreateProgramPayload, actorId: string): Promise<ProgramDefinitionRow>`
- `updateProgramDefinition(id: string, payload: UpdateProgramPayload, actorId: string): Promise<ProgramDefinitionRow>`
- `activateProgramDefinition(id: string, actorId: string): Promise<ProgramDefinitionRow>`
- `archiveProgramDefinition(id: string, actorId: string): Promise<ProgramDefinitionRow>`
- `linkOrgCatalogEntry(orgId: string, catalogEntryId: number, actorId: string): Promise<void>`
- `setUserProgramAffiliation(userId: string, catalogEntryId: number): Promise<void>`

Rules:
- All Supabase queries use the service-role client (imported from existing `lib/supabase/server.ts` or equivalent — confirm exact import path before writing)
- `activateProgramDefinition`: MUST deactivate any existing `is_active=true` record for the same `program_key` before activating target. Throw a typed `ServiceError` if target record is not found or not in `status='draft'`
- `createProgramDefinition`: sets `status='draft'`, `is_active=false`, `created_by=actorId`, `updated_by=actorId`
- `updateProgramDefinition`: only permitted on `status='draft'` records. Throws if record is `active` or `archived`
- `archiveProgramDefinition`: sets `is_active=false`, `status='archived'`, `updated_by=actorId`
- `linkOrgCatalogEntry`: validates `catalogEntryId` exists in the IL catalog (call `loadCatalog()` and check). Returns 422-equivalent error if not found. Checks for duplicate (another org already has the same `catalog_entry_id`) and returns 409-equivalent error if so
- `setUserProgramAffiliation`: upserts `profiles.affiliated_catalog_entry_id` for the given userId
- All mutations call `logEvent()` after success with appropriate action string
- All mutations call `updateProviderSearchIndexFromProgram()` after `logEvent()` — fire-and-forget (do not await if it would block the response, but do log errors)
- No inline role checks. Service is policy-agnostic; policy enforcement is the route's responsibility
- Use `applicant` (not `victim`) in all log messages, comments, error messages

Typed errors to define in this file (or `lib/server/programs/errors.ts`):
- `ProgramNotFoundError extends Error`
- `ProgramStateError extends Error` (wrong status for operation)
- `CatalogEntryNotFoundError extends Error`
- `CatalogEntryDuplicateError extends Error`
